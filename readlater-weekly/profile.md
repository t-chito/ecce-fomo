# 関心プロフィール

このファイルは「あとで読む」に保存した記事から読み取れる、現時点の関心と知識領域の地図である。時系列の記録ではなく、関心の構造を一枚で見渡すために随時書き換える。

## 中心にある関心: AIエージェントの運用設計 ―「回す人」を仕組みに置き換える

最も比重が大きい領域。かつては「エージェントに作らせる」ことが前提化し、関心は生成の前後の設計にあった。それがさらに一段上がり、いまの中心は**ループエンジニアリング**——人間が毎回プロンプトを打つ役から降り、エージェントを動かし続けるシステム全体を設計すること——にある。Prompt → Context → Harness → Loop と積み上がる4層の最上段を意識していて、下の各層への関心はこの上位目標に従属する部品として読めるようになっている。

この中心は次のように枝分かれしている。

- **完了の定義と外部検証**。「自己申告で完了にさせない」が繰り返し現れる急所。受け入れ基準を最初に固定し、test/lint/build の exit code という外部シグナルでのみ達成を判定する。同じ失敗を二度させない教訓テーブル、破壊的操作を止める制御、Small-Bet-First といった暴走ガードとセットで捉えている。生成と検証を別の役割に分ける以前からの関心が、ここに結実している。
- **手順と知識の注入、そして自己増殖**。プロジェクト固有の流儀をスキルや SOP、hook に切り出し、CLAUDE.md は毎回コンテキストに載る案内役に徹させて肥大化させない。さらに手順を書く作業自体をエージェントにやらせる自己増殖（自動生成スキルを Curator で整理する、会話ログから自分の設計哲学を抽出する）まで踏み込んでいる。「Claude がすでに知っていることは書かない」という抽出方針への共感は健在。
- **限界への醒めた目**。ループを賛美するだけでなく、自己改善エージェントが最初の前提を覆せず局所最適に留まる（アンカリング効果）という批判的考察を選んで拾う。仕組みを盲信せず、ハーネス層の設計で探索範囲を広げるという視点を併せ持つ。

ここから「トークン経済とコンテキスト効率」が枝分かれし、今週さらに具体化した。LLM はウィンドウに入っているものだけを見る以上、無関係な情報は精度そのものを下げる——だから need-to-know で絞ることはコスト削減と精度向上を同時に果たす。md2idx のような見出し単位の部分読み、オンデマンド読込、サブエージェントへの委譲による隔離が具体手段。圧縮率や削減効果を鵜呑みにせず /cost などで自分で再測定する慎重さも一貫している。

## 隣接する関心: AIに知識をどう渡し、どう引かせるか

エージェント運用と地続きで、AIに知識を構造化して渡す設計に関心が広がっている。Karpathy の LLM Wiki（人間ではなく LLM が wiki を書き、ingest で相互参照を同時更新し、lint で矛盾を検出し、複利的に成長させる）を起点に、その対極としての OKF（Markdown 冒頭に最小限のメタデータを付ける Google の規約）まで、自動生成と最低限の規律のバランスという構図で束ねて読んでいる。「全自動のナレッジ管理」の限界（どうでもいい情報を貯める・オレ流化する）への目配りもある。基盤側では Agentic retrieval のように、知識の「渡し方」だけでなく「引かせ方」の高度化（計画→複数回検索→引用付き合成）まで射程に入ってきた。

## 開発プロセスへの関心: 仕様駆動から意図駆動、そしてループへ

仕様駆動開発を追ううちに、仕様書への実装詳細の混入と二重管理の問題、その反作用としての意図駆動開発へ関心が分岐していた。この線は今、上記のループエンジニアリングへ合流している。「AIの出力を信頼するのではなく検証可能にする」という姿勢が、外部シグナルでの完了判定という形で具体化した。

## 基礎を固める関心: 設計原則を「AI出力の物差し」として使う

古典的な設計原則への関心は続いているが、その位置づけがはっきり変わった。いまは「知っておくべき教養」ではなく、**AIの大量出力を評価する物差し**として拾っている。SOLID や命名品質特性、Rails の責務分離は、レビューで言語化して指摘するための語彙を増やす道具。さらに「良い指示は良い設計語彙から生まれる」——原則を知っていることがプロンプトの解像度になる、という二次効果に強く反応している。知識の置き場所を How=コード / What=テスト / Why・Why not=ADR と4分類し「混ぜると全部腐る」とする整理は、この関心の現在の背骨。良し悪しの基準は OSS コードを大量に読むことでしか養われない、という地力論が土台にある。

## テストへの関心: 量から構造へ、振る舞いを一覧で保つ

テストを「正しさの確認」ではなく「品質を構造として設計し、人間にもAIにも渡す生きた仕様書」にする発想が、今週さらに具体化した。テストケースは観点のインスタンスに過ぎないと捉え、致命的不具合が異常系のごく一部に潜むという実データから、量への信仰を崩す。E2E を DSL から決定的にカタログ生成する、回帰テストをユーザーシナリオの幹に集約する（リグレッションベース）、変更からの伝播距離で機能テストと回帰テストを連続量として測る（テスト距離）といった、振る舞いを読める形で一覧に保つ手法に関心がある。AIがコードを書く量が増えるほど、外側の振る舞いを人間が責任を持って保証する重要性が上がる、という認識で一貫している。

## 新しく入ってきた関心: 速度から統制へ ― 開発を経営で捉える

これまで手薄だったプロダクト・ビジネスの視点に、今週はじめて意識的に手が伸びた。軸は「AIで作る速度が上がっても、事業のデリバリーや価値は速くならない」。GitLab 調査の「制御なき速度は負債」、アーキテクチャ改善を『Code Red』の定量データで経営投資として翻訳する試み、生成AIの影響は開発フェーズ（全体の10〜15%）に集中し運用が本体だとする指摘、開発生産性を量ではなく実現付加価値=レベル3で捉え直す議論（SaaS is Dead とも接続）。作り手の技術に閉じていた関心が、統制・価値・事業構造の側へ初めて開いた地点。

## フロントエンドと表現への関心: 制約を先に設計する

モダンCSS、レンダリング経路、キャッシュ階層、コンポーネント設計を「人間が制約を先に設計し、ブラウザやAIに判断を委ねる」という軸で追う関心は続いている。今週は SSR/CSR/SSG/ISR をデータ単位で使い分ける整理、React Profiler での計測、新HTTPメソッド QUERY の試しなど、手を動かす基礎固めが中心。生成AI（Claude Design）は画像でなくHTML/CSS/Reactの構造を返し既存資産からトーンを読み取る、という点に、デザインの土台を先に用意して出力の振れ幅を抑える発想との共鳴がある。ストーリー構成や視線誘導、そして「AIっぽさ」で受け手に忌避されないよう仕上げは自分の手で行う、という伝達設計は人間の領分に残す認識も持っている。

## 質への関心: 量より検証

AIで量を増やすこと自体には価値を置かず、人間が判断・検証できる形まで質を高めることに価値を置く。バージョン固定や実機・ソース照合、決定的チェックといった根拠の明示を伴うかで質を見分ける。この「量より検証」は、エージェント運用の外部検証への関心と同じ根を持つ。

## 周縁の関心

点在しつつ続く領域。基盤・セキュリティ（踏み台サーバーを attack surface で捉え直す、有料化した LocalStack の代替として MiniStack でローカルAWSを再現）、資格・学習法、ログ設計やデプロイ高速化、ローカルで動く生成AI。中心ほど深掘りしないが、技術の周辺にある実務的・社会的な論点にも目を配る。

## まだ拾えていない視点

作り手側への偏りは、経営・統制の視点が入ったことで一段和らいだ。一方でユーザーリサーチやマーケティングといった顧客解像度の側、AI活用の倫理・規制・コスト構造といったマクロな論点は依然として手薄。
