まとめ

この 2 週間の最前線には共通の型がある。名前のついた手法（SDD、ループ、スペック駆動、ソフトウェア工場）を万能視する言説に対して、実践者は「実験の帰結」と「適用条件」を持ち寄り始めた。t_wada は SpecKit を剥がして仕様書ソース化の限界を報告し、Armin Ronacher はハーネスが KV キャッシュを静かに燃やす仕組みを解説し、Dex Horthy は「人間レビュー抜きのソフトウェア工場」が壊れる理由を 1.5 万語で書いた。プロンプトの上手さはもう論点ではなく、検証・停止・キャッシュ・レビューといった制御系の工学が現在の水準線になっている。

今週の最前線

t_wada の「SpecKit 剥がし」— 仕様書をソースオブトゥルースにする実験の帰結

和田卓人は 7 月 10 日、「詳細な設計書からコードはいつでも再生成」という実験が上手くいかなかったと報告した。理由は明快で、「文書間の整合性を保つのはコードの整合性を保つより難しい」。帰結として、文書が不要なくらい綺麗なコードを書き（書かせ）、コードに書けないこと — Why と Why not — を文書に残し、全てをバージョン管理するスタイルに戻したという。この投稿は日本語圏で 2,000 近い like を集め、数日にわたる議論の起点になった。

続く整理がさらに具体的だ。Plan モードと SDD の違いは「作成された仕様書をどう扱うか」— Spec-first か、Spec-anchored/Spec-as-source か — に絞られ、後者は開発期間が長く分量が増えると綻びが出る（期待と逆）。だから「Plan モードで徹底的に議論し、ADR にしっかり残す」方針が手堅い、というのが現時点の結論。Jaana Dogan (rakyll) は関連する議論で「経験者は最初から全部をスペックしようとしない。反復こそがスキルであり、制約や発想が出てくるたびに実験してピボットする」と応じており、否定されているのは SDD の道具立てではなく「仕様書を正とする」運用だと分かる。

キャリブレーション: 自分の SDD 運用が Spec-first（議論の道具）か Spec-as-source（正本）かを区別する。後者に寄っているなら、最前線はすでに一周して戻ってきている。

  • t_wada: 詳細な設計書からコードはいつでも再生成という実験は上手くいかず
  • t_wada: Plan モードと SDD の違いは仕様書の扱い方に絞られる
  • rakyll: それこそがスキル。反復が前提

「プロンプトを書く仕事は終わった」— Boris Cherny とループエンジニアリング

Claude Code を作った Anthropic の Boris Cherny は「私はもう Claude にプロンプトしない。Claude にプロンプトするループを走らせていて、私の仕事はループを書くことだ」と語る。Peter Steinberger の 6 月 7 日の投稿「コーディングエージェントにプロンプトするのはもうやめるべきで、エージェントにプロンプトするループを設計すべきだ」は 24 時間で 500 万ビューに達し、Addy Osmani がこの実践を「ループエンジニアリング」と命名して用語が定着した。

中身は制御系の設計そのものだ。ループに必要なのはトリガー（PR が開く、スケジュール、人間の「go」）と検証可能なゴール（到達を機械的に判定できる終了状態）で、よくある失敗は 3 つ — メモリファイルが無く毎ループゼロから始まる、サブエージェント分割が無く 1 体に全部やらせる、停止条件が無く寝ている間に課金され続ける。無限ループ・ゴールドリフト・コスト爆発は edge case ではなく production の定番問題として扱われている。

キャリブレーション: 「良いプロンプトの書き方」はもう最前線の論点ではない。自分のループに記憶・分割・停止条件の 3 点が揃っているかが、そのままチェックリストになる。

  • Peter Steinberger: You shouldn't be prompting coding agents anymore

ハーネス内部の工学 — KV キャッシュを「静かに燃やさない」設計

Armin Ronacher (mitsuhiko) は、エージェントハーネスが KV キャッシュを助けているのか静かに燃やしているのかという議論を受けて、キャッシュの動作原理を解説する記事を公開した。要点: キャッシュはトークン列の前方一致でしか効かず、1 トークンの差分でそれ以降が全て無効になる。だからツールを会話の途中で外すとキャッシュは全滅する一方、追加は末尾差分で済む。タイムスタンプ入りの動的 system prompt は害で、コンパクションは「キャッシュのリセット」として意図的にやるべき操作になる。彼の Pi はこの原理に沿って「キャッシュを壊さない動的ツールロード」を実装し、キャッシュヒット率と再課金トークンを可視化した。

同じ週、Steinberger は Opus 4.8 が想定分布から外れた使い方で性能が落ちる観測に対し「システムと戦っても勝てない」として、自前ハーネスから公式 claude CLI を直接呼ぶコードパスを追加したと明かした。モデル単体の能力ではなく、モデルとハーネスの噛み合わせが実効性能を決めるという認識が、作っている側の共通見解になりつつある。Thorsten Ball の冷や水も添えておく — 毎日大量のトークンを使い、フロンティアモデルに早期アクセスしている本人ですら、リリース直後の精緻で断定的な分析を見るたびに「もう少し時間をかけてから言え」と思う、と。

  • mitsuhiko: agent harnesses are helping or quietly torching their caches
  • steipete: hard to fight the system
  • thorstenball: just give it more time

検証: 喧伝と実態

「skill を書けば資産が積み上がる」— 完成条件を誰も言えない

言説: エージェントに skill を書かせて自動蓄積すれば、組織の知識が資産として積み上がる。

実態: 「みんなエージェントの skill を書いているが、いつ完成と言えるかをほぼ誰も言えない」という指摘が 7 月に出た。skill の価値は中に入っている専門知識の質でしか決まらず、研究側でも one-shot で自動生成した skill は実行ベースの検証を通さない限り信頼性が出ないことが示されている（生成 skill を実行トレースで修正する SkillRevise のような研究が出てくること自体が、素の自動生成では足りない証拠でもある）。Simon Willison は Skills を「MCP より大きい」と評価するが、それは仕組みの評価であって、自動生成で中身が埋まるという話ではない。

較正: skill の価値は「書いた数」ではなく「検証済みの専門知識がどれだけ入っているか」。検証を通していない自動生成 skill は資産ではなく在庫。

  • Everyone Is Writing Skills for Their Agents. Almost Nobody Can Say When They're Complete.
  • SkillRevise: Improving LLM-Authored Agent Skills via Trace-Conditioned Skill Revision

「設計は賢いモデル、実装は安いモデル」— 適用条件が抜けている

言説: 設計・レビューを上位モデルに、実装を下位モデルに振り分ければコストと品質を両立できる。

実態: この分業は最前線でも実践されている — Simon Willison は「実装は適切な下位モデルをサブエージェントで走らせ、判断・レビュー・統合は main loop に残す。実装作業に最上位モデルが要ることは稀」と運用を公開している。ただし同じ週、日本語圏では「実装のほうが難易度が高い場合にやってはいけない」という指摘が流れ、t_wada が同意を示した。両者は矛盾していない。このルールの本質はコスト削減ではなく「判断をどこに残すか」であり、タスクの難所が実装側にあるなら分業の前提が崩れる。

較正: 手法名（モデル分業）ではなく適用条件（難所がどこか）で判断する。難所が設計にあるなら分業は効き、実装にあるなら逆効果。

  • simonwillison.net: Fable's judgement
  • t_wada: それな（実装のが難易度高い場合にやってはいけない）
