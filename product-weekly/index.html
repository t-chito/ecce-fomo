まとめ

今週の二本柱は「コーディング特化モデルの競争激化」と「Claude Code のメンテナンス機能の強化」。Claude Code は肥大化した skill・MCP・CLAUDE.md を掃除する /checkup を追加し、モデルと Effort レベルの使い分けを公式に整理した。外では SpaceXAI が「初のコーディング/エージェント特化」を掲げる Grok 4.5 を Cursor と共同学習で投入し、Cognition は 1000 tok/s の SWE-1.7 を Devin に載せた。加えて OpenAI が主要ベンチ SWE-Bench Pro の 30% が壊れているとする監査結果を公表し、推奨を撤回している。

Claude Code

/checkup（=/doctor）——増えすぎた skill・MCP・CLAUDE.md を棚卸しする

2.1.205 で /doctor が本格的なセットアップ点検コマンドに刷新され、/checkup はそのエイリアスとして案内された。bcherny が挙げた用途は、(1) 使っていない skill/MCP/plugin を整理してコンテキストを節約、(2) ローカルの CLAUDE.md と check-in 済み CLAUDE.md の重複を除去、(3) 肥大化した root CLAUDE.md を入れ子の CLAUDE.md と skill へ分割、(4) 遅い hook を無効化、(5) Claude Code 自体の更新——の5つ。

エージェントを長く使うほど設定とコンテキストが太り、パフォーマンスと精度を圧迫していく。その「棚卸し」を一発で回す運用ツールで、oikon48 も日本語で紹介している。地味だが、日常的に一番効く追加になりそうな機能。

モデルと Effort レベル——「賢さ」と「やり切る深さ」は別の軸

Anthropic が Claude Code のモデル選択と Effort レベルについての解説記事を公開した。ポイントは両者が独立した軸だという整理で、モデルは「頭の良さ・専門性」を決め、Effort レベルは思考時間だけでなく「読むファイルの量や検証の深さ＝どこまで作業を広げてやり切るか」を決める。

同じモデルでも Effort を上げれば調査と検証が深くなる、という現場の肌感覚が公式に言語化された形。モデルを上げるのか Effort を上げるのかで無駄なコストを避けられるので、使い分けの指針として押さえておきたい。

Fable 5 を「助言役」に——Sonnet 5 実行と組み合わせてコストを抑える

Fable 5 の全有料プラン開放が 7/12 まで延長されたのに合わせ、ClaudeDevs がコスト最適化の定番パターンを共有した。ひとつは Advisor パターンで、実行役の Sonnet 5 が要所で Fable 5 に助言を求める形。大半のトークンは安い実行役レートで課金される。もうひとつは Fable 5 が計画して Sonnet 5 ワーカーに委譲する Orchestrator パターンだ。

数字も出ている。SWE-bench Pro では「Sonnet 5＋Fable 5 アドバイザー」が Fable 5 単体の約 92% のスコアを約 63% のコストで達成（Fable 5 の呼び出しはタスクあたり概ね1回）。BrowseComp でも Orchestrator 構成が Fable 5 の 96% を 46% のコストで出したという。一方、Advisor 設定時にエラーへ遭遇する報告も出ており、セッションのコンテキストサイズが引き金と見られている。大きな文脈で使うときは注意したい。

2.1.203〜2.1.205——トランスクリプト改ざん防止と rm -rf の確認

直近リリースは安全側の作り込みが目立つ。2.1.205 では auto mode にセッション・トランスクリプトファイルの改ざんをブロックするルールを追加し、未解決の変数を含む rm -rf を実行する前に確認するようにした。ほかに --json-schema が不正スキーマ時に黙って非構造化出力を返す問題や、Windows の worktree 削除が NTFS junction／symlink 経由で worktree 外のファイルを消してしまう問題を修正。

2.1.203 ではログイン期限切れ前の警告や、バックグラウンドエージェント周りの多数の修正が入った。派手な新機能より、エージェントに任せたときの「事故らなさ」を積み上げる週だった。

Cowork がモバイル/Web へ——PC を閉じても走り続ける

Claude Cowork のモバイル・Web 版が発表された（Max プランを皮切りに数週かけてベータ提供）。デスクで渡したタスクの続きをスマホで受け取れるほか、スケジュールしたタスクは PC がオフでも実行される——朝6時に準備を走らせておけば、判断が要る所だけスマホに質問が来る、という使い方だ。

SWE-1.7 が Devin に——1000 tok/s の低コスト near-frontier

Cognition が最も高性能という SWE-1.7 を公開し、Devin（Web/Desktop/CLI）で即日利用可能にした。フロンティアモデルと数ポイント差の性能を大幅に安いコストで、しかも 1000 tok/s の「lightning モード」で出すという。Kimi K2.7 をベースに Devin ハーネス上で RL 学習し、長時間タスクで self-compact する設計だ。

挙動としては、編集前の調査・リサーチにより多くの時間を使うのが特徴。独自ベンチ FrontierCode（「実際にマージしたくなるコードか」を評価する）で cost-performance の Pareto を前進させたとしており、「速さ×安さ×マージ品質」を同時に狙う路線が続いている。

ベンチマークの信頼性

OpenAI が SWE-Bench Pro を監査——タスクの 30% が「壊れている」として推奨を撤回

OpenAI が、広く使われるコーディングベンチ SWE-Bench Pro を監査し、もはやフロンティア能力を確実には測れないとして、研究コミュニティへの推奨を撤回した。タスクの約 30% に問題があり、隠れた要件・矛盾する指示・厳しすぎるテスト・不完全な採点基準のせいで、正しい解が失敗してしまうと指摘する。監査にはモデルベースの調査エージェントと、5 名の独立した経験豊富なエンジニアのレビューを併用したという。

各社が「SWE-bench 系で何 %」と競う一方で、その物差し自体が歪んでいれば比較は成り立たない。コーディングモデルの進歩に合わせて、ベンチも harder / fairer / trustworthy にする必要がある——数字の受け取り方を一段慎重にさせる問題提起だ。
